# Runbook - migrations 0057 / 0058 / 0059 (PAGES + RECORDS)

The highest-risk step is the historical **DOCS -> RECORDS** conversion in
`0058_pages_records_units`. It rewrites existing `work_report_tasks` rows for
three exact DOC IDB sub-activities. **Never run it against production without
first completing the clone test in section 3.**

Three other DOC IDB sub-activities are genuinely document-based and must stay
on DOCS, which is why every lookup uses exact trimmed name matching - never a
prefix or `ILIKE`.

| Item | Value |
|---|---|
| Starting revision | `0056_work_items` |
| Target head | `0059_activity_req_pages_records` |
| Preflight | `backend/scripts/migration_preflight_0057_0059.py` (read-only) |
| Postflight | `backend/scripts/migration_postflight_0057_0059.py` (read-only) |

Set these once per shell:

```bash
export PGHOST=<prod-host> PGPORT=5432 PGUSER=<prod-user> PGDATABASE=<prod-db>
export STAMP=$(date +%Y%m%d-%H%M%S)
export PROD_URL="postgresql+psycopg://$PGUSER@$PGHOST:$PGPORT/$PGDATABASE"
export CLONE_DB="${PGDATABASE}_clone_${STAMP}"
export CLONE_URL="postgresql+psycopg://$PGUSER@$PGHOST:$PGPORT/$CLONE_DB"
```

---

## 1. Read-only preflight against production

Safe on a live database: it opens a `READ ONLY` transaction, runs SELECTs only
and rolls back.

```bash
python backend/scripts/migration_preflight_0057_0059.py \
    --database-url "$PROD_URL" \
    --out "preflight-prod-$STAMP.json"
echo "exit=$?"
```

**Gate.** Exit code must be `0`. It exits `1` when the revision is unexpected,
one of the three exact activities is missing, a sub-activity resolves
ambiguously, duplicate pending activity-requests exist (0057's unique index
would fail), or the historical counts are internally inconsistent.

Keep `preflight-prod-$STAMP.json` - the postflight needs it.

## 2. Back up production

```bash
pg_dump --format=custom --no-owner --no-privileges \
        --file="prod-$STAMP.dump" \
        --host="$PGHOST" --port="$PGPORT" --username="$PGUSER" "$PGDATABASE"

ls -lh "prod-$STAMP.dump"                 # size must be non-trivial
sha256sum "prod-$STAMP.dump" | tee "prod-$STAMP.dump.sha256"
pg_restore --list "prod-$STAMP.dump" | head    # dump must be readable
```

**Gate.** Non-zero file size, checksum recorded, `pg_restore --list` succeeds.
A dump you have not listed is not a backup you can trust.

## 3. Restore into a clone and rehearse the migration

```bash
createdb --host="$PGHOST" --port="$PGPORT" --username="$PGUSER" "$CLONE_DB"

pg_restore --no-owner --no-privileges --exit-on-error \
           --host="$PGHOST" --port="$PGPORT" --username="$PGUSER" \
           --dbname="$CLONE_DB" "prod-$STAMP.dump"

# sanity: clone matches production's starting revision
psql --host="$PGHOST" --username="$PGUSER" --dbname="$CLONE_DB" \
     -c "select version_num from alembic_version;"
```

Preflight the clone (must match the production preflight):

```bash
python backend/scripts/migration_preflight_0057_0059.py \
    --database-url "$CLONE_URL" --out "preflight-clone-$STAMP.json"

diff <(jq -S 'del(.captured_at,.database)' "preflight-prod-$STAMP.json") \
     <(jq -S 'del(.captured_at,.database)' "preflight-clone-$STAMP.json") \
  && echo "clone matches production"
```

Run the migrations against the clone only:

```bash
cd backend
DATABASE_URL="$CLONE_URL" alembic current
DATABASE_URL="$CLONE_URL" alembic upgrade head   # 0057 -> 0058 -> 0059
DATABASE_URL="$CLONE_URL" alembic current        # expect 0059_activity_req_pages_records
```

Verify the clone:

```bash
python backend/scripts/migration_postflight_0057_0059.py \
    --database-url "$CLONE_URL" \
    --preflight "preflight-clone-$STAMP.json"
echo "exit=$?"
```

**Gate.** Exit code `0`, every invariant PASS. In particular
`converted records_count total == preflight docs + records` proves each unit
moved exactly once - not dropped, not double-counted.

## 4. Generate a historical workbook from the clone

Proves migrated values now report under RECORDS and unrelated DOCS still report
under DOCS. Use an offset whose cycle contains the DOC IDB report dates printed
by the preflight (`report dates` line).

```bash
cd backend
DATABASE_URL="$CLONE_URL" python - <<'PY'
import os
from app.core.database import SessionLocal
from app.modules.benchmarks.service import get_pending_benchmark_export
from app.modules.reports_export.export import (
    build_pending_benchmark_workbook, date_range_label,
)

WEEK_OFFSET = int(os.environ.get("WEEK_OFFSET", "1"))
db = SessionLocal()
data = get_pending_benchmark_export(db, cycle=WEEK_OFFSET)
buf = build_pending_benchmark_workbook(data["rows"], data["cycle_start"], data["cycle_end"])
name = f"CLONE BENCHMARK REPORT {date_range_label(data['cycle_start'], data['cycle_end'])}.xlsx"
open(name, "wb").write(buf.getvalue())
print("wrote", name)
PY
```

Inspect the sheet (28 columns, A..AB):

- **RECORDS columns** `N` (target) / `T` (actual) / `Z` (pending) carry the three
  converted DOC IDB sub-activities.
- **DOCS columns** `J` / `P` / `V` still carry the unrelated DOCS activities,
  with the same totals as the preflight's `unrelated_docs.docs_total`.
- No converted row appears under DOCS.

**Gate.** Both conditions hold, and the DOCS totals equal the preflight value.

## 5. Production migration (only after every gate above passed)

Requires explicit approval and a maintenance window.

```bash
# stop writers so no report is submitted mid-migration
docker compose stop frontend worker beat

cd backend
DATABASE_URL="$PROD_URL" alembic current
DATABASE_URL="$PROD_URL" alembic upgrade head
DATABASE_URL="$PROD_URL" alembic current        # expect 0059_activity_req_pages_records

python scripts/migration_postflight_0057_0059.py \
    --database-url "$PROD_URL" \
    --preflight "preflight-prod-$STAMP.json"
echo "exit=$?"

docker compose up -d backend frontend worker beat
```

**Gate.** Postflight exit `0`. If it is non-zero, roll back (section 6) rather
than attempting a forward fix under time pressure.

## 6. Rollback

`0058` and `0059` deliberately **refuse to downgrade** - the DOCS -> RECORDS
move is not reversible in place, because after the fact a records value is
indistinguishable from one that was always records. Recovery is restore-only:

```bash
docker compose stop backend frontend worker beat

dropdb   --host="$PGHOST" --username="$PGUSER" "$PGDATABASE"
createdb --host="$PGHOST" --username="$PGUSER" "$PGDATABASE"
pg_restore --no-owner --no-privileges --exit-on-error \
           --host="$PGHOST" --username="$PGUSER" \
           --dbname="$PGDATABASE" "prod-$STAMP.dump"

psql --host="$PGHOST" --username="$PGUSER" --dbname="$PGDATABASE" \
     -c "select version_num from alembic_version;"   # expect 0056_work_items

docker compose up -d backend frontend worker beat
```

This is why section 2 is mandatory and why the dump is checksummed and listed
before anything is migrated.

## 7. Cleanup

```bash
dropdb --host="$PGHOST" --username="$PGUSER" "$CLONE_DB"
```

Retain `prod-$STAMP.dump`, its `.sha256`, and both preflight JSON files with the
release record.

---

## Decision gates at a glance

| # | Gate | Proceed only when |
|---|---|---|
| 1 | Production preflight | exit `0`, revision `0056_work_items` |
| 2 | Backup | non-zero size, checksum recorded, `pg_restore --list` OK |
| 3 | Clone preflight | matches production snapshot |
| 4 | Clone `alembic upgrade head` | reaches `0059_activity_req_pages_records` |
| 5 | Clone postflight | exit `0`, all invariants PASS |
| 6 | Clone workbook | converted rows under RECORDS, unrelated under DOCS |
| 7 | Approval + window | explicitly granted |
| 8 | Production postflight | exit `0`, else roll back per section 6 |
